# Product Roadmap — Architect Enterprise Builder

## Vision

사용자의 아이디어/니즈를 입력하면, 개발자가 즉시 MVP 프로토타입을 착수할 수 있는 수준의 백엔드 설계 문서를 자동 생성하는 AI 시스템.
Phase 1에서 백엔드 문서 품질을 극대화한 후, PO 관점에서 상용화를 위한 전체 프로덕트 플래닝으로 확장.

## Phase 1: 백엔드 설계 문서 품질 극대화 (현재 — 완료)

### 출력물 구조

**클라이언트 뷰 (비개발자용)**
- 문제 정의, 솔루션 개요, 핵심 기능, 마일스톤, 기대 효과, 데이터 보호, 투자 요약

**개발자 뷰 (4개 탭)**
- 로드맵: 스프린트 계획 (타임라인 연동)
- 아키텍처: 3개 Mermaid 다이어그램 (시스템/기술스택/시퀀스)
- 구현: 프로젝트 구조, 기술 스택, API 설계 (request/response/errorCodes), DB 스키마, 핵심 모듈 코드, 배포 계획, 테스트 전략
- 문서: PRD (8개 섹션), LLD (구체적 명세 형태)

**내보내기**
- HTML 보고서 (클라이언트/개발자 별도)
- ZIP 패키지 (개별 소스 파일 포함)
- JSON 원본
- 인쇄

## Phase 2: PO 관점 전체 프로덕트 플래닝 (계획)

### 목표

백엔드 설계 문서를 기반으로, 상용 프로덕트를 완성하기 위한 전체 영역의 설계 문서를 추가 생성.

### 확장 영역

#### 2-1. 프론트엔드 설계
- 화면 설계서 (와이어프레임/컴포넌트 구조)
- 페이지 라우팅 맵
- 상태 관리 설계 (전역/로컬)
- UI/UX 가이드라인 (디자인 시스템)
- 접근성(a11y) 요구사항

**구현 방안**: 현재 5단계 진단에 "프론트엔드 플랫폼" 질문 추가 (웹/모바일앱/하이브리드/어드민 대시보드). Claude 프롬프트에 프론트엔드 설계 섹션 추가.

#### 2-2. 서버 인프라 설계
- 클라우드 아키텍처 (AWS/GCP/Azure)
- CI/CD 파이프라인 상세 설계
- 환경별 구성 (dev/staging/prod)
- 스케일링 전략 (오토스케일링, 로드밸런싱)
- 모니터링/알림 구성 (Grafana, CloudWatch, Sentry)
- 비용 추정 (월간 인프라 비용)

**구현 방안**: Gemini 블루프린트에 인프라 설계 그룹(Group D) 추가. Context7으로 AWS/Docker/Kubernetes 공식 문서 레퍼런스 확장.

#### 2-3. 페이먼트 연동 설계
- PG사 연동 명세 (Stripe, Toss, 카카오페이 등)
- 결제 플로우 시퀀스 다이어그램
- 구독/일회성 결제 모델 설계
- 환불/취소 프로세스
- 정산 시스템 설계
- PCI-DSS 보안 요구사항

**구현 방안**: 진단 질문에 "결제 시스템 필요 여부" 추가. 해당 시 페이먼트 전용 프롬프트 활성화.

#### 2-4. 어드민 패널 설계
- 관리자 권한 체계 (RBAC)
- 대시보드 화면 구성
- 사용자/콘텐츠/주문 관리 CRUD
- 통계/분석 뷰
- 운영 도구 (공지, 배너, 설정)

**구현 방안**: LLD에 어드민 전용 섹션 추가. API 설계에 어드민 엔드포인트 그룹 자동 생성.

### 진단 질문 확장안

현재 5단계에서 8단계로 확장:

```
Phase 1: 비즈니스 배경 (기존)
Phase 2: 시스템 모델 (기존)
Phase 3: 프로세스 로직 (기존)
Phase 4: 기술 환경 (기존)
Phase 5: KPI 목표 (기존)
Phase 6: 프론트엔드 플랫폼 (NEW — 웹/앱/하이브리드/어드민)
Phase 7: 결제 시스템 (NEW — 필요 여부, PG사 선호, 결제 모델)
Phase 8: 어드민/운영 요구사항 (NEW — 관리 기능, 대시보드 필요 여부)
```

### 출력물 확장

개발자 뷰 탭 확장:
```
현재: 로드맵 | 아키텍처 | 구현 | 문서
확장: 로드맵 | 아키텍처 | 백엔드 | 프론트엔드 | 인프라 | 문서
```

## Phase 3: 워크플로우 자동화 (계획)

### 목표

생성된 문서를 기반으로 실무 프로젝트 관리까지 자동화.

### 기능

| 기능 | 설명 |
|------|------|
| Jira/Linear 연동 | 스프린트 계획 → 자동 이슈/태스크 생성 |
| GitHub 연동 | 프로젝트 구조 → 리포지토리 자동 초기화 |
| 개발자 온보딩 패키지 | 문서 + 코드 + 환경 설정을 하나의 패키지로 |
| 프로젝트 템플릿 | 기술 스택별 보일러플레이트 자동 생성 |
| 이메일/슬랙 공유 | 생성된 문서를 팀에 자동 배포 |

### 구현 방안

- 백엔드 서버 필요 (현재는 브라우저 전용)
- OAuth 연동 (GitHub, Jira, Slack)
- `services/emailService.ts`, `services/gdriveService.ts` 스텁 활용

## 기술 아키텍처 확장 노트

### 현재 아키텍처의 확장성

```
UI Layer (React) ← 탭/뷰 추가만으로 확장 가능
    ↓
Orchestration (useChat) ← Phase 추가만으로 확장 가능
    ↓
Services (Gemini/Claude) ← 프롬프트 섹션 추가로 확장 가능
    ↓
Stores (Zustand) ← 독립 스토어이므로 새 스토어 추가 용이
    ↓
Persistence (Dexie) ← 스키마 버전업으로 확장 가능
```

### Phase 3 구현 시 아키텍처 변경

```
현재: 브라우저 → Gemini/Claude API (직접 호출)
확장: 브라우저 → 백엔드 서버 → Gemini/Claude API
                    ↓
              GitHub/Jira/Slack API
              Email/GDrive
```

- 백엔드 서버 도입 필요 (Next.js API Routes 또는 Express)
- API 키를 서버로 이동 (보안 강화)
- OAuth 인증 추가

## 시장 포지셔닝

**타겟 사용자**
- 비개발자 창업자: 아이디어 → 개발 의뢰용 스펙 문서
- 사내 기획자/PM: 요구사항 → 개발팀 핸드오프 문서
- 소규모 팀: 외주 개발 의뢰 전 스펙 정리
- 프리랜서 개발자: 고객 요구사항 → 프로젝트 플래닝 자동화

**경쟁 차별화**
- Copilot/Cursor: 코드 생성에 집중 → 이 시스템은 "코드 작성 전 단계의 설계 문서"에 집중
- 기존 컨설팅: 고비용, 시간 소요 → AI 기반 즉시 생성
- 노코드 도구: 실행에 집중 → 이 시스템은 설계/문서에 집중
